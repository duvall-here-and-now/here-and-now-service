## Domain Model Architecture

### Document Types in Cosmos DB

All documents share a single container (`Tasks`) with `/userId` partition key and `type` discriminator:

| Document Type | Type Value | Purpose | Key Relationships |
|---------------|-----------|---------|-------------------|
| TaskDocument | `"Task"` | Regular tasks | → TaskReminderDocument via `reminderId` |
| TaskReminderDocument | `"TaskReminder"` | Time-based reminders | → TaskDocument via `taskId`, denormalized `taskName` |
| RecurringTaskConfigDocument | `"RecurringTaskConfig"` | Recurrence pattern definitions (RRULE) | Parent of state overrides |
| RecurringTaskStateOverrideDocument | `"RecurringTaskStateOverride"` | Explicit state changes for specific instances | → RecurringTaskConfigDocument via `configId` |

### Computed Model (Not Persisted)

| Model | Purpose | Generated By |
|-------|---------|-------------|
| RecurringTaskInstance | Represents a specific occurrence of a recurring task | `RecurringTaskService.ComputeInstances()` |

### State Machines

**Regular Tasks:**
```
OnDeck ──► InProgress ──► Completed
  │             │
  └─────────────┼──► Deleted (terminal, soft-delete)
                └──► OnDeck (revert)
```

**Recurring Task Instances:**
```
Scheduled ──► OnDeck ──► InProgress ──► Completed
                │             │
                └─────────────┼──► Skipped
                              └──► OnDeck (revert)
```

Rules:
- **One-active-at-a-time:** Only the most recent past instance can be OnDeck/InProgress per config
- **Older instances auto-skip:** Past instances without overrides become Skipped if a newer one is active
- **Terminal states respected:** Completed/Skipped overrides are never recalculated
- **Future = Scheduled:** All future occurrences are always Scheduled

## Unity Pattern (Atomic Operations)

The Unity pattern uses Cosmos DB transactional batches to atomically update multiple documents within the same partition:

```csharp
var batch = _container.CreateTransactionalBatch(new PartitionKey(userId));
batch.ReplaceItem(task.Id, task);
batch.ReplaceItem(reminder.Id, reminder);
var response = await batch.ExecuteAsync();
```

### Unity Operations

| Operation | Method | Documents Modified |
|-----------|--------|-------------------|
| Complete task | CompleteWithUnityAsync | Task (→Completed) + Reminder (→Dismissed) |
| Delete task | DeleteWithUnityAsync | Task (→Deleted) + Reminder (→Dismissed) |
| Update name | UpdateWithReminderSyncAsync | Task (name) + Reminder (taskName) |
| Create task+reminder | CreateTaskWithReminderBatchAsync | Task (create) + Reminder (create) |
| Create reminder | CreateWithTaskLinkAsync | Reminder (create) + Task (patch reminderId) |
| Delete config | DeleteConfigWithOverridesAsync | Config (delete) + all Overrides (delete) |

### Chunked Batch Deletion

For recurring config deletion with >99 overrides (Cosmos DB batch limit = 100):
1. Delete overrides in chunks of 100 (each chunk is a separate atomic batch)
2. Delete config document last
3. If config delete fails, orphaned overrides are acceptable (overrides without config are inert)

## Recurring Task Computation Architecture

The recurring task system uses a **computed instance model** — only configuration and explicit state overrides are persisted. Instances are computed on-the-fly:

```
┌──────────────────────┐     ┌─────────────────────────────┐
│ RecurringTaskConfig   │     │ RecurringTaskStateOverride    │
│ (RRULE, text, start) │     │ (configId, dateTime, state)  │
└──────────┬───────────┘     └──────────────┬───────────────┘
           │                                │
           ▼                                ▼
    ┌──────────────────────────────────────────┐
    │     RecurringTaskService.ComputeInstances │
    │                                          │
    │  1. Generate RRULE occurrences (Ical.Net)│
    │  2. Match overrides by composite key     │
    │  3. Apply one-active-at-a-time rule      │
    │  4. Assign default states (Scheduled,    │
    │     OnDeck, Skipped)                     │
    └──────────────────┬───────────────────────┘
                       │
                       ▼
            ┌──────────────────────┐
            │ RecurringTaskInstance  │
            │ (computed, ephemeral) │
            └──────────────────────┘
```

### Two-Query Pattern (NFR44)

`GetComputedInstancesAsync` makes exactly 2 database queries:
1. `GetAllConfigsAsync(userId)` — all configs for user
2. `GetStateOverridesForDateRangeAsync(userId, from, to)` — overrides in window

All computation happens in-memory after these two calls.

### Override Key Format

State overrides use a composite ID: `{configId}_{yyyy-MM-ddTHH:mm:ssZ}`
- Enables efficient lookup without additional queries
- Both dictionary key and override ID use the same `:O` (round-trip) format for consistency

### RRULE Validation

- Parsed via Ical.Net `RecurrencePattern`
- Rejected frequencies: Secondly, Minutely
- Accepted: Hourly, Daily, Weekly, Monthly, Yearly
- Stored without `RRULE:` prefix

- **Provider:** Auth0
- **Scheme:** JWT Bearer tokens
- **Validation:** Issuer, audience, and signing key validated


- `TaskNotFoundException` → 404
- `ReminderAlreadyExistsException` → 409
- `InvalidStateTransitionException` → 409
- `InvalidRecurrenceRuleException` → 400
- `RecurringTaskConfigNotFoundException` → 404

**SecureHeadersMiddleware** adds security headers:
- X-Content-Type-Options: nosniff
- X-Frame-Options: DENY
- Content-Security-Policy
- Strict-Transport-Security

- **Database:** HereAndNow
